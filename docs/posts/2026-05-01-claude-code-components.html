Claude Codeの仕組みを解説：CLAUDE.md・エージェントチーム・hooks・memoryの役割

agentaiworkflowtutorial

Claude Codeを構成する6つのコンポーネント（CLAUDE.md・agent teams・skills・rules・hooks・memory）の役割と連携を、実際のリポジトリ構成をもとに初心者向けに解説します。

はじめに

Claude Codeは、Anthropicが提供するAIアシスタントをコマンドラインやエディタから使えるツールです。単純に「質問に答えてもらう」だけでなく、プロジェクトのルールを覚えさせたり、複数のAIエージェントに役割を分担させたりと、高度な自動化ワークフローを組めるのが特徴です。

この記事では、Claude Codeを構成する主要な6つのコンポーネントを、このブログ（agent-blog）の実際の構成ファイルを参照しながら解説します。「なんとなく使い始めたけど、.claude/ フォルダの中身が何なのかよくわからない」という方に向けた内容です。

> 初心者向けまとめ: Claude Codeは「設定ファイル＋役割分担AIチーム」で動くツールです。本記事を読むと、その仕組みの全体像が見えてきます。

CLAUDE.md — AIへの「仕事の引き継ぎ書」

CLAUDE.md は、プロジェクトのルールや構成をClaude Codeに伝えるための設定ファイルです。Claudeはセッション開始時にこのファイルを自動的に読み込み、プロジェクトの文脈を把握したうえで作業を開始します。

たとえば、このブログの CLAUDE.md には次のような内容が書かれています。

  • ブログのディレクトリ構成（posts/・drafts/・sources/ など）
  • 記事フォーマット（frontmatter の書き方、必要なセクション）
  • コーディング規約（Markdownのみ編集する、ファイル名のルールなど）
  • 絶対ルール（git push --force の禁止、posts/ への直接書き込み禁止など）

仕事でいうと、「新しいアルバイトさんに渡すマニュアル冊子」のようなものです。毎回同じ説明をしなくても、CLAUDE.md さえ読めばプロジェクトの進め方がわかります。

さらに CLAUDE.local.md というファイルも使えます。こちらは個人の作業スタイルや環境メモなど、チームには共有しない個人設定を書く場所です。.gitignore に追加しておけばリポジトリには含まれません。

> 初心者向けまとめ: CLAUDE.md は「このプロジェクトでは何をどうやって作るか」をAIに伝える最重要ファイルです。まずここを丁寧に書くと、AIの動きが格段に安定します。

agent teams — 役割分担したAIチームの仕組み

Claude Codeでは、複数のAIエージェントに異なる役割を持たせて協力させる「エージェントチーム」を構成できます。エージェントとは、特定の目的に特化した設定を持つAIの動作単位のことです。エージェントの定義は .claude/agents/ フォルダに保存します。

このブログでは以下の4つのエージェントが連携しています。

producer（オーケストレーター）
    ├── researcher  — 最新情報をWebから収集する
    ├── writer      — 収集結果をもとに記事を執筆する
    └── editor      — 記事をSEO・文体の観点でレビューする

各エージェントはMarkdownファイル（例：researcher.md）で定義されていて、次の項目を持ちます。

  • name / description: エージェントの名前と役割の説明
  • tools: そのエージェントが使えるツール（WebFetch, Read, Write など）
  • プロンプト本文: 起動時に何をどの順でやるかの詳細な手順書

たとえば researcher.md には「Step 1: メモリを読み込む」「Step 2: Tier 1サイトをWebFetchで確認する」「Step 6: COLLECT_RESULT形式で報告する」といった手順が書かれています。producer（オーケストレーター）がresearcherを起動し、返ってきた結果を確認してからwriterを起動する、という直列パイプラインが組まれています。

仕事でいうと、「編集長（producer）が取材担当（researcher）・執筆担当（writer）・校閲担当（editor）をそれぞれ専門チームに振り分けて進行管理する編集部」のイメージです。

> 初心者向けまとめ: エージェントチームを使うと、AIが「収集→執筆→レビュー」のような複数ステップの作業を自律的にこなせるようになります。人間は最終確認だけすればよい状態が作れます。

skills と rules — 「手順書」と「守るべきルール集」

skills: 作業の手順書

.claude/skills/ フォルダには、各コマンド（/collect・/draft・/review・/publish）の詳細な手順書（SKILL.md）が入っています。

たとえば /collect の SKILL.md には「Step 1: 収集対象を決める」「Step 2: Tier 1の4サイトをWebFetchで確認する」「Step 3: 候補が3件未満なら補足検索する」といったステップが定義されています。AIはこの手順書を読んで作業を進めるため、毎回同じ品質で情報収集を実行できます。

rules: 文体・SEO・情報源のルール

.claude/rules/ フォルダには、AIが記事を書くときに守るべきルールが3つのファイルに分かれて保存されています。

  • tone.md: 「ですます調を使う」「中学生でもわかる言葉を選ぶ」「禁止表現の一覧」など文体のルール
  • seo.md: タイトルの文字数・見出し構成・frontmatterの書き方などSEO（検索エンジン最適化）のルール
  • sources.md: 信頼できる情報源のリスト（Tier 1〜3の分類と収集フォーマット）

skills が「何をどうやるか（手順）」を定義するのに対して、rules は「何をしてはいけないか・何に従うか（制約と基準）」を定義します。料理でいえば、skills は「レシピの手順」で、rules は「食品衛生の基準と使っていい食材のリスト」に当たります。

> 初心者向けまとめ: skills で作業手順を標準化し、rules で品質基準を統一することで、誰が（どのエージェントが）作業しても同じクオリティの結果が得られます。

hooks と memory — 「自動ガード」と「学習する記憶」

hooks: 危険な操作を自動でブロックするガード

hooks（フック）とは、特定のタイミングに自動実行されるプログラムのことです。.claude/hooks/ フォルダには、シェルスクリプト形式のhooksが入っています。

  • pre_tool_use.sh: ツール実行前に走るスクリプト。git push --force の禁止、posts/ への直接書き込みのブロック、拡張子チェックなどを実施します。
  • session_start.sh: セッション開始時に走るスクリプト。未完成の下書き一覧・記事化候補件数・公開待ちファイルの一覧を自動表示します。

hooksは「交通信号機」のようなものです。AIが危険な操作（例：git push --force）をしようとしたとき、人間が気づく前に自動でブロックします。セッション開始時のhooksは「今日の作業状況の朝礼報告」として機能し、何をすべきかをすぐに把握できます。

memory: 経験を蓄積するエージェントの記憶

.claude/agent-memory/ フォルダには、各エージェントが学習・蓄積する情報が保存されます。

  • researcher の source_quality.md: 過去に有効だった情報源・避けるべき情報源の実績
  • writer の covered_topics.md: 執筆済み記事のトピック一覧（重複防止用）
  • writer の writing_patterns.md: レビューを通過した記事構成パターン
  • editor の common_issues.md: 繰り返し起きているレビュー指摘事項

たとえば writerエージェントは記事を書く前に covered_topics.md を参照し、すでに似た記事を書いていないか確認します。またresearcherは作業後に source_quality.md を更新し、今回使えた/使えなかった情報源を記録します。これにより、エージェントは実績を積むほどより良い判断ができるようになります。

仕事でいうと、「先輩社員が『これは使えた、これは失敗だった』と引き継ぎノートを書き、後任が同じ失敗を繰り返さないようにする仕組み」です。

> 初心者向けまとめ: hooks は「ルール違反を自動で防ぐ仕組み」、memory は「AIが経験から学ぶ仕組み」です。この2つがあることで、AIは使うほど賢くなり、ミスも減っていきます。

まとめ

Claude Codeの主要コンポーネントを整理すると、次のようになります。

| コンポーネント | 場所 | 役割 | |---|---|---| | CLAUDE.md | プロジェクトルート | プロジェクト全体のルール・構成の説明 | | agents/ | .claude/agents/ | 各エージェントの役割・手順・使えるツールの定義 | | skills/ | .claude/skills/ | 各コマンドの詳細な作業手順書 | | rules/ | .claude/rules/ | 文体・SEO・情報源などの品質基準 | | hooks/ | .claude/hooks/ | 危険操作のブロックと作業状況の自動表示 | | agent-memory/ | .claude/agent-memory/ | エージェントが経験から学ぶ記憶の保存先 |

これらは単独でも機能しますが、組み合わせることで大きな力を発揮します。CLAUDE.md でプロジェクトを理解し、rules で品質を保ち、skills で手順を標準化し、hooks でミスを防ぎ、memory で経験を蓄積する。このサイクルが回ることで、AIが「使い捨てのアシスタント」から「成長するチームメンバー」へと変わっていきます。

Claude Codeを使い始めたばかりの方は、まず CLAUDE.md を丁寧に書くことから始めてみてください。それだけでもAIの動きは大きく変わります。次のステップとして、.claude/rules/tone.md のような文体ルールファイルを追加すると、アウトプットの品質がさらに安定します。Anthropic の公式ドキュメントも合わせて参照してみてください。
