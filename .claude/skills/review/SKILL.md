# 記事レビューワークフロー

## 目的

下書き記事をレビューし、公開前の品質チェックを行う。ファイルは**書き換えない**。フィードバックのみ返す。

## 手順

### Step 1: 対象ファイルを特定する

$ARGUMENTS があればそのファイルを対象にする。
なければ `drafts/` の中で最終更新が新しいファイルを対象にする。

### Step 2: frontmatterチェック

以下を確認する:

- [ ] `title` が30〜50字か
- [ ] `date` が正しい形式（YYYY-MM-DD）か
- [ ] `tags` が2〜4個か、既存タグを使っているか
- [ ] `summary` が1〜2文・80字以内か

### Step 3: 文体チェック（tone.md 準拠）

- [ ] ですます調で統一されているか
- [ ] 禁止表現（「革命的」「〜という感じ」など）が含まれていないか
- [ ] 技術用語に初出解説があるか
- [ ] 1段落が4〜6文程度か

### Step 4: 構成チェック（seo.md 準拠）

- [ ] H2見出しが2〜4個か
- [ ] 冒頭200字で記事の内容が明示されているか
- [ ] まとめセクションがあるか
- [ ] 外部リンクが2〜5個か

### Step 5: 内容チェック

- [ ] 事実確認できない情報が含まれていないか
- [ ] ソースが `sources/collected.md` に記載されているか
- [ ] AIが生成した「それらしいが不正確な情報」がないか（架空のツール名・誤った日付・存在しないURLなど）

### Step 5a: 文字数チェック

本文の文字数を確認する（frontmatter・参考情報行を除いた本文のみ）:

- [ ] 本文が **1000字以上** か
- 800字未満の場合は「修正必須」、800〜999字は「修正推奨」とする

### Step 5b: 著作権・引用チェック

- [ ] ソース記事からの引用が **1箇所あたり100字以内** に収まっているか
- [ ] 引用箇所には必ず出典URL が付いているか
- [ ] 記事の大部分がソースのコピーになっていないか（独自の要約・解説が半分以上あるか）

### Step 5c: 重複トピックチェック

`.claude/agent-memory/writer/covered_topics.md` を参照し:

- [ ] 直近3ヶ月以内に同じトピックを取り上げた記事がないか
- 重複している場合は「修正推奨」として新しい切り口の追加を提案する

### Step 5d: 個人情報・機密情報チェック

- [ ] 実在する個人のフルネーム・連絡先・所在地が含まれていないか
- [ ] 内部システム名・APIキー・トークン類が含まれていないか
- [ ] 「要確認」マークのついた未検証情報が残っていないか

### Step 6: フィードバックを出力する

以下の形式で出力する:

```
## レビュー結果: [ファイル名]

### 合格項目 ✓
- （問題なかった項目を列挙）

### 修正推奨 ⚠️
- （修正した方がよい箇所：行番号と理由）

### 修正必須 ✗
- （公開前に必ず直すべき箇所：行番号と具体的な修正案）

### 総合評価
公開可能 / 要修正（修正後に再レビュー推奨）
```

### Step 7: レビューステータスを書き込む

総合評価が **「公開可能」** の場合のみ、対象ファイルの frontmatter に
`status: reviewed` を追加（または更新）する。

- `status:` 行がすでにある場合 → `status: reviewed` に書き換える
- ない場合 → frontmatter の最終フィールドの直後（closing `---` の前）に追加する

この書き込みにより、`/publish` スキルがレビュー済みであることを自動検証できる。
**「要修正」判定の場合は書き込まない。**
