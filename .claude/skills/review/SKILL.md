# 記事レビューワークフロー

## 目的

下書き記事をレビューし、公開前の品質チェックを行う。ファイルは**書き換えない**。フィードバックのみ返す。

## 手順

### Step 1: 対象ファイルを特定する

$ARGUMENTS があればそのファイルを対象にする。
なければ `drafts/` の中で最終更新が新しいファイルを対象にする。

### Step 2: frontmatterチェック

以下を確認する:

- [ ] `title` が30〜50字か
- [ ] `date` が正しい形式（YYYY-MM-DD）か
- [ ] `tags` が2〜4個か、既存タグを使っているか
- [ ] `summary` が1〜2文・80字以内か

### Step 3: 文体チェック（tone.md 準拠）

- [ ] ですます調で統一されているか
- [ ] 禁止表現（「革命的」「〜という感じ」など）が含まれていないか
- [ ] 技術用語に初出解説があるか
- [ ] 1段落が4〜6文程度か

### Step 4: 構成チェック（seo.md 準拠）

- [ ] H2見出しが2〜4個か
- [ ] 冒頭200字で記事の内容が明示されているか
- [ ] まとめセクションがあるか
- [ ] 外部リンクが2〜5個か

### Step 5: 内容チェック

- [ ] 事実確認できない情報が含まれていないか
- [ ] ソースが `sources/collected.md` に記載されているか
- [ ] AIが生成した「それらしいが不正確な情報」がないか

### Step 6: フィードバックを出力する

以下の形式で出力する:

```
## レビュー結果: [ファイル名]

### 合格項目 ✓
- （問題なかった項目を列挙）

### 修正推奨 ⚠️
- （修正した方がよい箇所：行番号と理由）

### 修正必須 ✗
- （公開前に必ず直すべき箇所：行番号と具体的な修正案）

### 総合評価
公開可能 / 要修正（修正後に再レビュー推奨）
```
